## ADDED Requirements

### Requirement: 설정 메뉴 허브
설정 탭의 root 화면은 11개 카테고리의 메뉴 목록만 표시하는 허브 화면이어야 한다. 각 메뉴 항목은 카테고리 이름과 간단한 설명/현재 상태 표시(예: "알람 3개")를 포함하며, 탭 시 해당 디테일 스크린으로 push 네비게이션한다.

#### Scenario: 허브 진입 시 메뉴 목록 표시
- **WHEN** 사용자가 BottomTab의 "설정" 탭을 누른다
- **THEN** 시스템은 11개 카테고리(알람 설정, 바다 관리, 가족 공유, 화면 모드, 홈화면 구성, AI 태그 관리, 백업/복원, 동기화 진단, 데이터/프라이버시, 후원, 앱 정보)를 메뉴 행으로 표시한다
- **AND** 각 행은 좌측 라벨, 우측 chevron(`›`) 아이콘을 포함한다

#### Scenario: 메뉴 항목 탭 시 디테일 스크린 진입
- **WHEN** 사용자가 메뉴 행 중 하나를 탭한다
- **THEN** 시스템은 해당 디테일 스크린으로 push 전환한다
- **AND** 헤더에 카테고리 이름이 타이틀로 표시된다
- **AND** 헤더 좌측에 OS 기본 뒤로가기 버튼이 표시된다

### Requirement: 디테일 스크린 뒤로가기
모든 디테일 스크린은 OS 표준 뒤로가기(헤더 화살표 + iOS 스와이프 백 + Android 시스템 백)로 허브로 복귀할 수 있어야 한다.

#### Scenario: 헤더 뒤로가기 버튼
- **WHEN** 사용자가 디테일 스크린에서 헤더의 뒤로가기 화살표를 탭한다
- **THEN** 시스템은 허브 화면으로 복귀한다
- **AND** 허브의 스크롤 위치는 진입 직전 상태로 유지된다

#### Scenario: iOS 스와이프 백 제스처
- **WHEN** 사용자가 iOS에서 디테일 스크린의 좌측 가장자리에서 우측으로 스와이프한다
- **THEN** 시스템은 허브 화면으로 복귀한다

#### Scenario: Android 시스템 백
- **WHEN** 사용자가 Android에서 시스템 백 버튼/제스처를 사용한다
- **THEN** 시스템은 허브 화면으로 복귀한다

### Requirement: 진행 중 작업 보호
백업/복원/AI 태그 재분석처럼 사용자가 시작한 비동기 작업이 진행 중인 디테일 스크린에서는 사용자가 무심코 뒤로가기하여 작업이 중단되지 않도록 보호한다.

#### Scenario: 백업 진행 중 뒤로가기 시도
- **WHEN** 백업이 진행 중(`isBackingUp = true`)인 상태에서 사용자가 뒤로가기를 시도한다
- **THEN** 시스템은 확인 다이얼로그("백업 중입니다. 정말 나가시겠어요?")를 표시한다
- **AND** 사용자가 "취소"를 선택하면 화면에 머문다
- **AND** 사용자가 "나가기"를 선택하면 백업 작업을 취소하지 않고 허브로 복귀한다(작업은 백그라운드 계속)

#### Scenario: 일반 상태에서 뒤로가기
- **WHEN** 진행 중 작업이 없는 상태에서 사용자가 뒤로가기를 시도한다
- **THEN** 시스템은 확인 없이 즉시 허브로 복귀한다

### Requirement: 메뉴 허브의 상태 요약 표시
허브의 각 메뉴 행은 해당 카테고리의 현재 상태를 간단히 요약 표시하여 디테일 진입 없이도 핵심 정보를 파악할 수 있어야 한다.

#### Scenario: 알람 설정 행 요약
- **WHEN** 사용자가 허브 화면에서 "알람 설정" 행을 본다
- **THEN** 활성 알람이 있으면 "N개 활성", 없으면 "비활성" 같은 간략한 부가 텍스트가 우측 chevron 옆에 표시된다

#### Scenario: 바다 관리 행 요약
- **WHEN** 사용자가 허브 화면에서 "바다 관리" 행을 본다
- **THEN** 등록된 바다 수 또는 활성 바다 이름이 부가 텍스트로 표시된다

#### Scenario: 동기화 진단 행 요약
- **WHEN** 동기화 실패 또는 대기 중인 항목이 있다
- **THEN** "확인 필요" 같은 경고 표시가 부가 텍스트로 표시된다

### Requirement: 디테일 스크린 간 이동 금지
디테일 스크린 간에는 직접 횡 이동하지 않고 항상 허브를 거쳐서 이동해야 한다(허브-디테일 트리 구조 유지).

#### Scenario: 디테일에서 다른 디테일로 이동
- **WHEN** 사용자가 한 디테일 스크린에서 다른 디테일 스크린으로 가려고 한다
- **THEN** 시스템은 디테일 → 허브 → 다른 디테일 순서로 이동을 강제한다 (디테일 스크린 내에서 다른 디테일로의 직접 링크 제공 금지)

### Requirement: 디테일 스크린 내부 상태 자율 관리
각 디테일 스크린은 자신의 입력/모달/진행 상태를 자체 관리하며, 허브나 다른 디테일 스크린과 상태를 공유하지 않는다.

#### Scenario: 알람 추가 모달이 알람 스크린 내부에 존재
- **WHEN** 사용자가 SettingsAlarmScreen에서 "+ 알람 추가"를 누른다
- **THEN** 시간 피커 모달이 SettingsAlarmScreen 내부 컴포넌트로 표시된다
- **AND** 다른 디테일 스크린에서 시간 피커 모달이 노출되지 않는다

#### Scenario: 바다 이름 변경 모달이 바다 관리 스크린 내부에 존재
- **WHEN** 사용자가 SettingsChildrenScreen에서 바다를 길게 눌러 이름 변경을 시도한다
- **THEN** 이름 변경 모달은 SettingsChildrenScreen 내부 컴포넌트로 표시된다

### Requirement: 공통 UI 컴포넌트
모든 디테일 스크린과 허브는 일관된 시각 스타일을 위해 공통 컴포넌트(`SettingsRow`, `SettingsSection`, `SettingsCard`)를 사용한다.

#### Scenario: 메뉴 행 일관성
- **WHEN** 사용자가 허브 또는 디테일 스크린에서 메뉴/설정 행을 본다
- **THEN** 모든 행은 동일한 패딩, 폰트 크기, chevron 아이콘 위치, 터치 피드백을 가진다

#### Scenario: 섹션 헤더 일관성
- **WHEN** 디테일 스크린이 여러 섹션으로 나뉘어 있다
- **THEN** 각 섹션 헤더는 동일한 폰트 크기·색상·여백을 가진다

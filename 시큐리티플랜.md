## 1. 점검 범위 합의

### 위협 모델 (먼저 정해야 우선순위가 정해짐)

| 위협 주체 | 시나리오 | 현재 대응 |
|---|---|---|
| **임의 외부 공격자** | APK 분해 → Worker secret 추출 → API 비용 폭주 | 분당 30 rate limit (IP별) |
| **단말 분실/도난** | 잠금 해제된 폰에서 기록 열람 | OS 기본 암호화 |
| **루팅/탈옥 단말** | SQLite 파일 직접 추출 | 없음 (앱 샌드박스 의존) |
| **악의적 가족 구성원** | 초대코드 무차별 → 타가족 데이터 접근 | Supabase RLS (서버 점검 필요) |
| **개발자 실수** | `.env` 커밋, 키스토어 푸시 | gitignore 적용됨 |
| **Vendor (Google/Groq)** | 전송 데이터 학습/보존 | paid tier 정책 (확인 필요) |

→ **레드라인**: 비용 폭주, RLS 우회로 타인 데이터 접근, 키스토어 유출
→ **수용 가능**: 루팅 단말 SQLite 노출 (사용자 책임 영역)

## 3. 2차 점검 영역 (다음 세션)

각 영역은 **별도 세션**으로 분리해 컨텍스트 오염 방지.

### A. 서버 측 (이 레포 외부)
- **Supabase RLS 정책 점검** — `records`, `family_members`, `families` 테이블의 row-level security가 user_id 기준으로 강제되는가
- **Storage bucket 정책** — `photos` 버킷이 private이고 signed URL 외 접근 차단되는가
- **Supabase rate limit / abuse 설정** — 익명 가입 무제한 여부
- **Cloudflare/Deno 환경변수** — 서버 콘솔에서 secret 회전 가능 여부

### B. 의존성/공급망
- `npm audit` 결과 검토
- `expo-image-picker`, `expo-file-system/legacy` 등 deprecation 경로 확인
- Supabase JS SDK 버전 + CVE
- Whisper/Gemini SDK 버전

### C. 클라이언트 추가 점검
- **딥링크/URL scheme 검증** — `Linking.addEventListener` 받는 URL 검증 강화 ([AppNavigator.tsx:118](src/navigation/AppNavigator.tsx#L118))
- **권한 최소화** — 카메라/마이크/사진 라이브러리 권한 요청 시점이 명확한가
- **로그/console.warn 누설** — 운영 빌드에서 민감 데이터 콘솔 출력 여부
- **에러 메시지 누설** — alert에 내부 경로/스택 노출 여부
- **AsyncStorage/FileSystem 평문 설정 파일** — `app_settings.json`에 민감값 없는지 재확인

### D. 운영/프로세스
- 키 회전 절차 문서화
- pre-commit secret 스캔 hook 도입 (gitleaks 등)
- CI에서 EAS secret 노출 방지 점검

## 4. 다음 점검 시 진행 방식

1. **위협 모델부터 사용자와 합의** — 무엇을 막을지가 정해져야 우선순위가 결정됨
2. **영역별 분리 점검** (A → B → C → D 순서 권장)
3. **각 영역마다 산출물**:
   - 발견 사항 (증거: 파일:라인)
   - 위협 모델 대비 실 위험도 평가 (외부 AI 보고서처럼 무조건 High로 분류 X)
   - ROI 기반 우선순위 (수정 비용 vs 위험 감소량)
4. **수정은 별도 세션** — 점검과 수정을 섞으면 판단이 흐려짐

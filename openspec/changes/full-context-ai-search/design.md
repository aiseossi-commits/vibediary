## Context

현재 AI 등대는 RAG(Retrieval-Augmented Generation) 파이프라인으로 동작한다. 사용자 질문을 임베딩으로 변환 → 벡터 유사도로 관련 기록 선별 → LLM 답변 생성. 이 구조는 의미론적으로 유사한 기록을 찾는 데는 동작하지만, "3월에 무슨일이 있었지?"처럼 날짜·집계·패턴 기반 질문에서 유사도 게이트에 막혀 답변 불가 상태가 된다.

Gemini 2.5 Flash Lite의 컨텍스트 윈도우는 1M 토큰이며, 이 앱의 전체 기록(5년 사용 기준 ~5000건 × 100자 = 약 15만 토큰)은 이 한도 내에 충분히 들어온다.

## Goals / Non-Goals

**Goals:**
- 날짜, 집계, 패턴, 의미론적 질문 모두 하나의 파이프라인으로 처리
- 기록 추가 시 임베딩 API 호출 제거 (비용·지연 감소)
- 검색 파이프라인 단순화 (벡터 검색·유사도 게이트·키워드 검색 제거)

**Non-Goals:**
- DB의 embedding 컬럼 제거 (스키마 변경 리스크 없이 NULL 유지)
- 검색 이외 화면(Home, Calendar, Tags)의 변경
- 컨텍스트 캐싱, 서버사이드 처리

## Decisions

### 결정 1: Full-context vs NL2SQL

NL2SQL도 날짜·집계 쿼리를 처리할 수 있으나, SQL 생성 오류 핸들링·스키마 노출·의미론적 쿼리 한계가 남는다. Full-context 방식은 LLM이 직접 모든 기록을 보고 판단하므로 어떤 질문 유형이든 단일 파이프라인으로 처리 가능하다. 이 앱의 데이터 규모에서는 컨텍스트 윈도우 한계에 도달하지 않으므로 Full-context를 선택한다.

### 결정 2: 기록 직렬화 포맷

LLM에게 전달할 때 토큰을 최소화하면서 날짜·태그·내용이 모두 읽히도록:

```
2026-03-15 #투약#일상 약 복용함
2026-03-10 #의료#투약 류코보린 20mg 증량함 [drug:류코보린,dose:20mg]
```

- 날짜: `YYYY-MM-DD` (날짜 기반 질문 처리를 위해 연도 포함)
- 태그: 붙여쓰기 (`#태그1#태그2`)
- structured_data: `#의료` 또는 `#투약` 태그 있는 기록만 `[key:val]` 포함 (토큰 절약)

### 결정 3: 컨텍스트 크기 상한

기록이 극단적으로 많아질 경우(2000건 초과)를 대비해 최근 2000건으로 제한한다. 2000건 × 평균 80자 ≈ 16만자 ≈ 4만 토큰으로 안전 범위 내.

### 결정 4: 시스템 프롬프트에 현재 날짜 포함

"3월에", "이번달", "지난주" 같은 상대적 날짜 표현을 LLM이 해석할 수 있도록 시스템 프롬프트에 `오늘: YYYY-MM-DD` 를 항상 포함한다.

### 결정 5: 임베딩 관련 코드 정리 범위

- `generateEmbedding()` 호출 제거: recordPipeline, offlineQueue, SettingsScreen 재색인 버튼
- `vectorSearch.ts` 파일 삭제
- `searchPipeline.ts` 전면 재작성
- DB embedding 컬럼: NULL 유지 (마이그레이션 불필요)
- `getRecordsWithEmbeddings()` 삭제

## Risks / Trade-offs

- [기록 급증 시 응답 지연] → 2000건 상한 + compact 포맷으로 완화. 실제 도달까지 수년 소요.
- [쿼리당 토큰 비용 증가] → Gemini Flash Lite는 저가($0.1/1M). 1회 검색 = 약 4만 토큰 = $0.004. 허용 범위.
- [임베딩 컬럼 데이터 잔존] → 기존 기록의 embedding 값이 DB에 남아있으나 더 이상 읽지 않으므로 무해.
- [오프라인 검색 불가] → 기존 방식도 LLM 호출이 필요했으므로 변화 없음.

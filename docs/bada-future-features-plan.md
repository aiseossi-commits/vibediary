## 1. 사진 첨부

### 개요

음성 기록에 사진을 첨부해, 행동·이벤트를 시각적으로 보완하는 기능.  
검색(AI 등대) 시 해당 기록과 함께 사진이 노출되는 것이 목표.

---

### 핵심 결정 1: 저장 방식

#### Option A. 갤러리 경로(URI)만 저장

갤러리에서 선택한 사진의 URI를 SQLite에 저장. 실제 파일은 갤러리에 그대로 남음.

**장점**
- 구현이 단순하고 빠름
- 앱 용량 증가 없음

**단점**
- 갤러리에서 사진 삭제 시 기록에서 사진이 깨짐 (참조 무결성 없음)
- 공유/내보내기 시 경로가 다른 기기에서 무의미해짐
- Android URI 만료 이슈 (권한이 앱 재설치 시 초기화될 수 있음)

**적합한 경우:** 빠른 MVP 검증, 단기 사용 위주 유저

---

#### Option B. 앱 내부 디렉토리에 복사 저장 ✅ 권장

`expo-file-system`으로 선택한 사진을 앱 전용 디렉토리(`FileSystem.documentDirectory/photos/`)에 복사 후 경로 저장.

**장점**
- 갤러리 삭제와 무관하게 기록 유지
- 백업 ZIP에 사진 포함 가능
- 공유 시 파일 패키징 가능
- 기록 삭제 시 사진도 함께 정리 가능

**단점**
- 기기 저장 용량 사용 (사진 1장 평균 2~5MB)
- 용량 관리 로직 필요 (사진 정리, 용량 경고 등)

**적합한 경우:** 장기 기록, 공유 기능 연동, 신뢰성 우선

---

#### Option C. 썸네일 + 원본 분리 저장

원본은 갤러리에 남기고, 리사이즈된 썸네일(최대 800px)만 앱 내부에 저장.

**장점**
- 용량 절감 (썸네일 100~300KB 수준)
- 표시 속도 빠름

**단점**
- 원본 갤러리 삭제 시 썸네일만 남음 (화질 제한)
- 구현 복잡도 증가 (이미지 리사이즈 처리 필요)
- 원본 공유 불가

**적합한 경우:** 용량 민감한 유저가 많을 경우

---

### 핵심 결정 2: 첨부 시점

#### Option A. 녹음 완료 후 첨부

녹음 → AI 정리 완료 → 기록 상세 화면에서 사진 추가.

**장점**
- 기존 녹음 UX 변경 없음
- 구현 범위 최소화

**단점**
- "말하면서 찍은 사진"과 기록이 분리되는 느낌

---

#### Option B. 녹음 중 또는 시작 전 첨부 ✅ 권장

녹음 화면에 사진 첨부 버튼 추가. 녹음과 사진을 한 번에 묶어서 저장.

**장점**
- 사진과 음성이 같은 맥락에서 생성됨
- UX 흐름이 자연스러움

**단점**
- 녹음 화면 UI 변경 필요
- 첨부 후 녹음 취소 시 사진 처리 로직 필요

---

### 핵심 결정 3: 탐색/검색 연동

사진 자체를 AI가 분석해서 임베딩하는 방식은 **비용과 복잡도가 과도**하므로 제외.

**채택 방향:** 사진은 기록(record) 단위에 묶임. AI 등대 검색 시 해당 기록이 조회되면 첨부 사진도 함께 노출.

즉, 사진을 직접 검색하는 게 아니라 **"기록을 찾으면 사진이 따라오는" 구조**.

---

### DB 변경 사항

```sql
-- records 테이블에 컬럼 추가
ALTER TABLE records ADD COLUMN photo_paths TEXT DEFAULT NULL;
-- JSON 배열 문자열로 저장: '["path/to/photo1.jpg", "path/to/photo2.jpg"]'

-- 또는 별도 테이블로 분리 (다대일 관계)
CREATE TABLE record_photos (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  record_id INTEGER NOT NULL REFERENCES records(id) ON DELETE CASCADE,
  file_path TEXT NOT NULL,
  created_at TEXT NOT NULL
);
```

**컬럼 추가 vs 별도 테이블 비교**

| 항목 | 컬럼 추가 | 별도 테이블 |
|------|----------|------------|
| 구현 단순도 | 높음 | 낮음 |
| 사진 개수 제한 | 없음 (JSON 배열) | 없음 |
| 삭제/정리 로직 | 직접 파싱 필요 | CASCADE로 자동 |
| 확장성 | 낮음 | 높음 |

사진이 1~3장 수준이면 컬럼 추가로 충분. 다중 첨부·정렬이 필요하면 별도 테이블 권장.

---

### 백업 JSON 변경

```json
{
  "records": [
    {
      "id": 1,
      "content": "...",
      "photo_paths": ["photos/abc123.jpg", "photos/def456.jpg"]
    }
  ],
  "photos": [
    {
      "filename": "abc123.jpg",
      "data": "<base64 인코딩>"
    }
  ]
}
```

사진을 base64로 인코딩해 JSON에 포함하면 단일 파일로 완전한 백업 가능.  
단, 파일 크기가 커지므로 ZIP 포맷으로 전환하는 것도 검토 필요.

---

### 용량 관리 정책 (필요 시)

- 앱 설정에서 현재 사진 총 용량 표시
- 기록 삭제 시 연결된 사진 파일도 함께 삭제
- 선택적으로 "사진 없이 백업" 옵션 제공

## 2. 공유 기능

### 개요

엄마·아빠 등 복수 양육자가 같은 아이의 기록을 함께 볼 수 있게 하는 기능.  
**서버 없이, 로컬 우선 철학을 유지**하는 방향으로 설계.

현재 바다에는 이미 JSON 백업/복원이 구현되어 있어, 이를 기반으로 확장하는 것이 자연스러움.

---

### 핵심 결정 1: 동기화 방식

#### Option A. 실시간 클라우드 동기화

Firebase Realtime DB, Supabase 등 서버를 통해 두 기기가 항상 같은 상태 유지.

**장점**
- 사용자 경험이 가장 매끄러움
- 어느 기기에서든 최신 기록 확인 가능

**단점**
- 서버 구축 및 유지 비용 발생
- 인증 시스템(로그인) 필요 → 현재 바다와 철학 충돌
- 개인정보처리방침 대폭 수정 필요
- 충돌(conflict) 처리 로직 복잡
- 솔로 개발자 유지보수 부담 극대화

**결론:** 현 단계에서 채택 불가

---

#### Option B. 주기적 수동 내보내기/가져오기 ✅ 단기 권장

한 명이 JSON으로 내보내기 → 카카오톡 등으로 전송 → 상대방이 가져오기.

**장점**
- 서버 불필요
- 기존 백업 기능 그대로 활용
- 구현 추가 작업 최소화
- 개인정보 리스크 없음

**단점**
- 수동 과정 필요 (불편함)
- 병합(merge) 시 충돌 처리 어려움
- 양쪽이 동시에 기록하면 한쪽 데이터 유실 가능성

**충돌 처리 방안:**
- 단방향 덮어쓰기: 가장 단순하나 데이터 유실 위험
- 타임스탬프 기반 병합: 더 최근 기록을 우선
- 중복 제거 후 합산: record id + created_at 기준으로 union

---

#### Option C. 마스터 기기 + 읽기 전용 공유 ✅ 중기 권장

한 기기(주 양육자)가 마스터. 다른 기기는 "읽기 전용 스냅샷"을 받아보는 구조.

```
마스터 기기 (엄마)          보조 기기 (아빠)
     기록 작성
        ↓
  JSON 내보내기  ──────────→  가져오기 (읽기 전용)
                              열람만 가능, 수정 불가
```

**장점**
- 충돌 원천 차단 (쓰기 권한 단일화)
- 구현 단순
- 현실적으로 많은 가정에서 "주 기록자"가 한 명인 경우가 많음

**단점**
- 아빠도 기록하고 싶을 경우 사용 불가
- "읽기 전용" 상태를 앱 내에서 표시하는 UI 필요

---

#### Option D. 로컬 네트워크(Wi-Fi) 직접 동기화

같은 Wi-Fi에 연결된 두 기기가 직접 통신해 기록을 동기화.

**장점**
- 서버 없이 실시간에 가까운 동기화 가능
- 데이터가 외부로 나가지 않음

**단점**
- 구현 복잡도 매우 높음 (Bonjour/mDNS, WebSocket 등)
- 같은 Wi-Fi 환경 필수 → 외출 중 동기화 불가
- Expo에서 네이티브 네트워크 모듈 추가 필요

**결론:** 기술적으로 가능하나 현 단계 비용 대비 효과 낮음

---

### 핵심 결정 2: 내보내기 포맷

#### Option A. JSON (현행 유지)

**장점:** 이미 구현됨, 파싱 쉬움  
**단점:** 사진 포함 시 파일 크기 폭증, 비기술 사용자에게 낯선 포맷

#### Option B. ZIP ✅ 사진 첨부 기능 추가 시 권장

JSON + 사진 파일을 묶어서 ZIP으로 내보내기.

**장점:** 사진 포함 가능, 단일 파일로 완전한 백업  
**단점:** `expo-file-system` + ZIP 라이브러리 추가 필요

#### Option C. PDF 리포트

기록을 읽기 좋은 PDF로 변환해 공유.

**장점:** 치료사·의사에게 전달하기 좋음, 가독성 높음  
**단점:** 편집/복원 불가, 단방향 공유만 가능  
**적합한 경우:** 공유 목적이 "함께 기록"이 아니라 "전문가에게 보여주기"일 때

---

### 공유 UX 흐름 (Option B + C 조합 권장)

```
공유 버튼 탭
    ↓
공유 목적 선택
  ├─ [가족과 함께 보기]  → ZIP 내보내기 (복원 가능)
  └─ [치료사·병원에 보내기] → PDF 리포트 생성
```

---

### 향후 클라우드 연동 준비

지금 내보내기 포맷(JSON 스키마)을 잘 설계해두면, 나중에 서버를 붙일 때 마이그레이션이 쉬워짐.

```json
// 공유 메타데이터 추가 권장
{
  "export_version": "1.1",
  "exported_at": "2026-04-07T10:00:00Z",
  "device_id": "uuid-익명",
  "child_id": "uuid",
  "records": [...]
}
```

`device_id`를 익명 UUID로 포함해두면 나중에 충돌 병합 시 출처 추적 가능.

## 3. 특별 이벤트 / 미션판

### 개요

변비, 수면, 식사량처럼 며칠~몇 주 단위로 추적해야 하는 항목을 빠뜨리지 않고 기록하도록 돕는 기능.  
핵심은 **노출** — 앱을 열었을 때 "오늘 이거 기록했어?" 하고 자연스럽게 상기시켜 주는 것.

---

### 용어 정의

내부적으로 "미션"보다 **"추적 항목(Tracker)"** 이라는 이름이 더 적합할 수 있음.  
"미션"은 게임적 뉘앙스가 있어 돌봄 맥락과 다소 어울리지 않을 수 있음.  
최종 네이밍은 사용자 피드백 후 결정 권장.

---

### 핵심 결정 1: 항목 등록 방식

#### Option A. 직접 입력

사용자가 항목 이름, 추적 단위, 종료 조건을 직접 입력.

```
항목 이름: 배변 여부
추적 단위: 매일
종료 조건: 2주 후 자동 종료 / 수동 종료
기록 방식: 체크 / 메모 / 음성 연동
```

**장점:** 유연함, 어떤 상황에도 적용 가능  
**단점:** 첫 사용 시 설정 허들 높음

---

#### Option B. 템플릿 선택 + 커스텀 ✅ 권장

자주 쓰이는 항목을 템플릿으로 제공하고, 없으면 직접 추가.

```
템플릿 예시:
  🚽 배변 여부 (매일)
  🌡️ 체온 (필요 시)
  💊 투약 여부 (매일)
  😴 수면 시간 (매일)
  🍽️ 식사량 (매일)
  + 직접 추가
```

**장점:** 진입 장벽 낮음, 빠른 시작 가능  
**단점:** 템플릿 기획 작업 필요, 업데이트 시 템플릿 관리 필요

---

### 핵심 결정 2: 기록 방식

#### Option A. 탭 한 번 체크 (Yes/No)

오늘 배변 있었나? → ✅ / ❌

**장점:** 가장 빠르고 단순  
**단점:** 세부 정보 기록 불가 (몇 번? 상태는?)

---

#### Option B. 간단 입력 (수치/선택지)

배변 횟수: [1회] [2회] [3회 이상]  
상태: [양호] [불량]

**장점:** 의미 있는 데이터 축적 가능  
**단점:** UI 설계 복잡도 증가

---

#### Option C. 음성 기록 연동 ✅ 권장 (바다의 핵심 UX 유지)

미션판 항목을 탭하면 → 음성 녹음 화면으로 연결  
녹음 내용이 해당 추적 항목과 자동 연결되어 저장.

```
[배변 여부] 탭
    ↓
녹음 화면 열림 (미션 컨텍스트 표시: "배변 여부 기록 중")
    ↓
"오늘 오전에 한 번 봤어요. 상태는 괜찮았어요."
    ↓
AI 정리 + 배변 항목 기록으로 저장
```

**장점:** 바다의 핵심 가치(음성 기록)와 자연스럽게 통합  
**단점:** 빠른 체크만 하고 싶은 경우 음성이 과할 수 있음

**절충안:** 탭 시 선택 가능 — [✅ 빠른 체크] [🎙️ 음성으로 기록]

---

### 핵심 결정 3: 노출 방식

#### Option A. 홈 화면 상단 배너

활성 추적 항목 중 오늘 미기록 항목을 홈 화면 상단에 표시.

```
┌─────────────────────────────┐
│ 오늘 아직 기록 안 한 항목    │
│ 🚽 배변 여부   💊 투약 여부  │
└─────────────────────────────┘
```

**장점:** 앱 열자마자 눈에 띔  
**단점:** 항목 없을 때 빈 공간 처리 필요, 홈 UI 변경 필요

---

#### Option B. 별도 탭 ✅ 권장

하단 탭바에 미션판 전용 탭 추가.

**장점:** 홈 UI 변경 최소화, 기능 영역 명확히 분리  
**단점:** 탭 추가 시 전체 네비게이션 구조 변경 필요

---

#### Option C. 뱃지(Badge) 알림

탭 아이콘에 미기록 항목 수 뱃지 표시.

**장점:** 비침습적, 사용자가 원할 때만 확인  
**단점:** 뱃지만으로 충분한 동기부여가 되는지 불확실

---

#### Option D. 푸시 알림

매일 특정 시간에 "오늘 배변 기록 하셨나요?" 알림.

**장점:** 앱을 열지 않아도 상기시켜 줌  
**단점:** future-features.md에 알림 기능이 UX 피로로 제거된 전례 있음. 재도입 신중하게 결정 필요.

---

### 핵심 결정 4: 종료 조건

| 방식 | 장점 | 단점 |
|------|------|------|
| 날짜 지정 (2주 후 자동 종료) | 잊어버릴 걱정 없음 | 상황이 바뀌면 수동 연장 필요 |
| 수동 종료 | 유연함 | 종료를 깜빡할 수 있음 |
| 목표 달성 시 종료 (예: 7일 연속 정상) | 의미 있는 완료 | 목표 설정 로직 복잡 |

**권장:** 날짜 지정 기본 + 수동 종료 병행. 목표 달성 방식은 추후 검토.

---

### DB 변경 사항

```sql
-- 추적 항목 테이블
CREATE TABLE trackers (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  child_id INTEGER NOT NULL REFERENCES children(id) ON DELETE CASCADE,
  name TEXT NOT NULL,                    -- "배변 여부"
  unit TEXT NOT NULL,                    -- "daily" | "asneeded"
  record_type TEXT NOT NULL,             -- "check" | "voice" | "number"
  is_active INTEGER NOT NULL DEFAULT 1,  -- 0: 종료됨
  started_at TEXT NOT NULL,
  ends_at TEXT,                          -- NULL이면 수동 종료
  created_at TEXT NOT NULL
);

-- 추적 기록 연결 (records 테이블과 연결)
ALTER TABLE records ADD COLUMN tracker_id INTEGER REFERENCES trackers(id);

-- 또는 간단 체크용 별도 테이블
CREATE TABLE tracker_checks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  tracker_id INTEGER NOT NULL REFERENCES trackers(id) ON DELETE CASCADE,
  checked_at TEXT NOT NULL,              -- "2026-04-07"
  value TEXT,                            -- "yes" | "no" | 수치
  note TEXT,                             -- 간단 메모
  record_id INTEGER REFERENCES records(id) -- 음성 기록 연결 시
);
```

## 4. structured_data 표준화 (선행 조건)

### 왜 필요한가

현재 AI가 추출하는 `structured_data`는 자유형식이라 기록마다 필드명이 다름.  
미션판에서 "배변 오늘 있었나"를 집계하거나, 웹 리포트에서 체온 그래프를 그리려면 **일관된 스키마**가 필수.

---

### 권장 표준 스키마

```json
{
  "temperature": 38.5,
  "medications": [
    { "name": "해열제", "dose": "1정", "time": "09:00" }
  ],
  "behavior_incident": true,
  "behavior_notes": "자해 행동 3회",
  "sleep_hours": 7.5,
  "sleep_quality": "poor | fair | good",
  "meal_amount": "적음 | 보통 | 많음",
  "bowel_movement": true,
  "bowel_notes": "정상",
  "tracker_data": {
    "tracker_id": 3,
    "value": "yes"
  }
}
```

---

### 변경 영향 범위

| 항목 | 내용 |
|------|------|
| AI 프롬프트 | structured_data 출력 형식 명시 필요 |
| 기존 기록 | 소급 적용 불가 (기존 기록은 자유형식 유지) |
| 신규 기록 | 표준 스키마로 저장 |
| 웹 리포트 | 표준화 이후 데이터부터 의미 있는 차트 가능 |

**결론:** 빠를수록 좋음. 미션판 구현 전에 반드시 완료.

## 5. 기능 간 연관 관계

```
structured_data 표준화
        ↓
   미션판 (tracker)
        ↓                    ↘
   사진 첨부             공유 기능
        ↓                    ↓
   record 단위에         ZIP / PDF
   사진 묶임             내보내기

   세 기능 모두 record 테이블 중심으로 연결됨
```

### 데이터 구조 최종 확장 방향

```
records 테이블
  ├── photo_paths (사진 첨부)
  ├── tracker_id  (미션판 연결)
  └── structured_data (표준화 스키마)

trackers 테이블 (신규)
tracker_checks 테이블 (신규)
record_photos 테이블 (선택적 신규)
```
